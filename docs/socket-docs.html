Transport Strategy & Delivery Guarantees

Transport Fallbacks

Default: WebSocket (wss://).
Fallback: HTTP long-polling.
Note: React Native clients strictly force WebSockets to prevent polling latency overhead on mobile networks.

Event Acknowledgements (ACK)

Fire-and-forget is disabled for critical mutations. Client-to-server events (e.g., room joins) implement callback ACKs to guarantee delivery and trigger retry logic if failed.

Idempotency & Flood Control

The server enforces idempotency. Redundant events caused by network retries (e.g., joining a room twice) are deduplicated. Strict per-socket rate limiting prevents event-loop blocking under high concurrency.

Security & Connection Lifecycle

Handshake Authentication

Connections are rejected at the TCP level if the RS256 JWT payload is missing or invalid. Tokens are verified once during the handshake, completely isolating the socket layer from unauthenticated traffic.

Payload Validation & Anti-Spoofing

Client payloads are strictly sanitized. User identities (e.g., userId, userName) are attached to the socket instance server-side, preventing clients from spoofing messages as other users.

Heartbeat (Ping/Pong)

A continuous 25s ping/pong interval guarantees connection health. Stale connections (e.g., ghost users who hard-closed the app) are aggressively swept and purged from presence maps.

Handling Distributed Failure Scenarios

Reconnect Storms (DDoS)

If the backend restarts, 10,000 clients attempting to reconnect instantly will crash the Auth service. Mitigation: Clients must implement randomized exponential backoff (reconnectionDelayMax: 5000).

Network Drop Mid-Game

Mobile users frequently traverse tunnels. Mitigation: Server buffers events for 2 minutes via connectionStateRecovery. If downtime exceeds 2 minutes, the client falls back to game:state:sync.

Message Ordering Desync

TCP guarantees order per-connection. However, across distributed shards, global order is lost. Mitigation: Future scale-outs require Redis Pub/Sub integration to guarantee chronologically consistent broadcasting.

Architectural Limitations & Scaling

In-Memory State Bottleneck

Currently, Room Presence and Chat History are mapped in process memory. This represents a critical bottleneck. If deployed behind a load balancer to multiple instances, state will become fragmented (User A on Instance 1 will not see User B on Instance 2).

The Migration Path

To support horizontal scaling, the architecture must implement @socket.io/redis-adapter for cross-node event propagation. In-memory presence maps must be migrated to Redis Hashes, and Chat Buffers to Redis Lists with TTL expiry.
